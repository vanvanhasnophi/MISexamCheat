# MIS 手绘考点总览：建模三件套 → UML → 结构化工具

---

## 一、建模三件套（核心手绘考点）

MIS 考试中最基础、最常考的三类建模图。

### 1. DFD 数据流图（Data Flow Diagram）

**四种基本符号：**

| 符号 | 图形 | 含义 |
|:---|:---|:---|
| Process（过程） | 圆角矩形 | 对数据的变换/处理 |
| Data Flow（数据流） | 箭头 → | 数据在系统内的流动路径 |
| Data Store（数据存储） | 开口矩形（或无右侧边的矩形） | 数据的暂存/持久化位置 |
| External Entity（外部实体） | 正方形或矩形 | 系统外的人/组织/系统 |

**基本规则：**
- 至少有一个 Process
- 不能有独立对象（所有对象必须通过数据流连接）
- Process 必须有输入和输出数据流
- Data Store 必须连接到至少一个 Process
- External Entity 之间不能直接连接

**自顶向下开发方法：**
```
上下文图（Context Diagram）
  编号 0，一个 Process，所有 External Entity + 主要数据流
        ↓
图0（Diagram 0）
  展开上下文图，最多 9 个 Process，引入主要 Data Store
        ↓
子图（Child Diagrams）
  编号 = 父 Process 编号，逐级展开
  原子过程（Primitive Process）= 不再展开
        ↓
检查错误 → 逻辑 DFD → 物理 DFD → 分区（Partitioning）
```

![Context.png](blob:https://blogmanager.eightyfor.top/15eccaca-36c2-49da-820d-0c6c0fcdfebd "上下文图")


![DFD_L0.png](blob:https://blogmanager.eightyfor.top/bbf2eb9d-0c27-41a6-91ee-8b2acea194d7 "L0 DFD")


![DFD_child.png](blob:https://blogmanager.eightyfor.top/113ce66f-5123-4e16-9f7d-71ade8434b9a "子 DFD")


**逻辑 DFD vs 物理 DFD：**
- 逻辑 DFD：业务如何运作（业务活动）
- 物理 DFD：系统如何实现（程序、模块、手工流程、物理文件）

**事件建模：** 外部实体的输入流 = 触发器（Trigger），启动 Process。事件响应表 → DFD 片段 → 单个 Process。

**参考：** §06 p37, §08 p18–p25

### 2. ERD 实体关系图（Entity-Relationship Diagram）

**核心要素：**

| 要素 | 图形/表示 | 说明 |
|:---|:---|:---|
| Entity（实体） | 矩形 | 需要追踪的人/地点/事物 |
| Attribute（属性） | 椭圆或列在实体中 | 实体的特征 |
| Relationship（关系） | 菱形或连线 | 实体间的业务关联 |
| Primary Key（主键） | 下划线 | 唯一标识每个实例 |

**三种关系多重性：**

| 关系 | 示例 |
|:---|:---|
| 一对一（1:1） | 一个学生有一个学号 |
| 一对多（1:M） | 一个客户有多笔订单 |
| 多对多（M:N） | 一个学生选多门课，一门课有多个学生 |


![image.png](blob:https://blogmanager.eightyfor.top/47233340-feac-497c-95f8-bcc62bf128af)


**破解多对多：** 引入关联实体（Associative Entity），将 M:N 拆为两个 1:M。

**与 DFD 的区别：**
- DFD：动态建模（数据如何流动和变换）
- ERD：静态建模（数据结构是什么）

**参考：** §06 p37

### 3. 用例图（Use Case Diagram）

**核心要素：**

| 要素 | 图形 | 说明 |
|:---|:---|:---|
| Actor（参与者） | 火柴人 | 系统外部与系统交互的角色 |
| Use Case（用例） | 椭圆 | 系统提供的一项功能 |
| System Boundary（系统边界） | 矩形框 | 系统的范围 |

**四种关系：**

| 关系 | 符号 | 含义 |
|:---|:---|:---|
| 通讯（Communicates） | Actor ─── Use Case | 参与者能使用该功能 |
| 包含 `<<include>>` | A - -→ B（虚线箭头） | A **必须**包含 B，B 被复用 |
| 扩展 `<<extend>>` | B - -→ A（虚线箭头） | B 是 A 的**可选**扩展 |
| 泛化（Generalizes） | 子 ──▷ 父（空心三角） | 继承关系 |


![用例图示例](blob:https://blogmanager.eightyfor.top/016b9543-5def-4209-9f36-0473ef5e6b73)

**用例场景模板（Use Case Scenario）：**

| 栏目 | 内容 |
|:---|:---|
| 用例名称 + 唯一ID | 功能名 + 编号 |
| 参与者（Actor） | 主参与者 / 次参与者 |
| 触发事件 | 什么触发此用例 |
| 执行步骤（主路径） | 1. 2. 3. ... 逐步描述 |
| 前置条件 | 执行前必须满足的状态 |
| 后置条件 | 执行后系统的状态 |
| 假设 / 未解决问题 | 约束和待确认项 |
| 优先级 / 风险 | 高/中/低 |


**参考：** §06 p37, §10 p11–p14

## 二、UML 图型详细拆解

### 4. 用例描述（Use Case Description）

**输入数据：**
- 用例名称和编号
- 业务规则（BR）
- 非功能性需求（NF）

**结构表格：**

| 栏目 | 输入内容类型 | 示例 |
|:---|:---|:---|
| 用例名称 | 功能名称 + 编号 | UC02 提交维修报修申请 |
| 参与者 | 角色清单 | 主：司机；次：调度员 |
| 前置条件 | 执行前状态 | 司机已登录、车辆故障 |
| 主流程 | 正常路径步骤 | 1.打开→2.拍照→3.提交→4a/4b/4c分支 |
| 扩展流程 | 异常处理 | E1无网络、E2照片无效 |
| 后置条件 | 执行后状态 | RepairRequest已创建、Vehicle已更新 |
| 关联需求 | 需求追溯 | BR01、BR02、NF01、NF02、NF03 |

**作用：** 补充用例图无法表达的流程细节，是活动图和序列图的直接输入源。

### 5. 活动图（Activity Diagram）

**七种基本符号：**

| 元素 | 图形 | 说明 |
|:---|:---|:---|
| 开始节点 | ● 实心圆 | 流程起点 |
| 活动节点 | 圆角矩形 | 操作描述 |
| 控制流 | 箭头 → | 活动间流转 |
| 决策/合并节点 | ◇ 菱形 | 分支判断 |
| 泳道（Swimlane） | 纵向/横向长矩形 | 参与者职责分区 |
| 分叉/汇合条 | 粗黑线 | 并行活动的拆分和同步 |
| 结束节点 | ◉ 实心圆套空心圆 | 流程终点 |

![活动图示例](blob:https://blogmanager.eightyfor.top/0fd904d0-d947-457d-b6b4-91212a403543)

**何时使用活动图：**
- 理解用例的活动流程
- 控制流复杂需要可视化
- 需要建模工作流
- 需要展示用例的所有场景

**注意事项：** 每个决策出口必须有路径；每个事件和决策都必须可测试以验证状态转换。

### 6. 类图（Class Diagram）

#### 6.1 核心类图（领域模型）

**类框结构（三格）：**

| 格子 | 内容 | 示例 |
|:---|:---|:---|
| 第一格 | 类名（斜体=抽象类） | `RepairRequest` |
| 第二格 | 属性（-私有/+公开/#保护） | `- requestId: String` |
| 第三格 | 方法 | `+ submit(): void` |

**关系类型速查：**

| 关系 | 符号 | 含义 |
|:---|:---|:---|
| 依赖 | ----→（虚线箭头） | 临时使用 |
| 关联 | ─────（实线） | 长期结构关系 |
| 聚合 | ──◇──（空心菱形） | 整体-部分，部分可独立 |
| 组合 | ──◆──（实心菱形） | 整体-部分，部分不可独立 |
| 泛化 | ──▷──（空心三角） | 继承关系 |
| 自反关联 | 从类出发回到自身 | 同类的对象间关系 |

**多重性标注：**

| 符号 | 含义 |
|:---|:---|
| 1 | 恰好一个 |
| * | 零到多个 |
| 0..1 | 零或一个 |
| 1..* | 一个到多个 |
| n..m | n 到 m 个 |

**四种类的类型：** 实体类 / 边界类（接口类）/ 控制类 / 抽象类

![类图示例](blob:https://blogmanager.eightyfor.top/7b24b1a5-915d-4f82-9dd4-ce491618745d)

#### 6.2 三层架构类图

| 层 | 包含 | 职责 |
|:---|:---|:---|
| 表示层（Presentation） | UI 组件、API Controller | 与用户/外部系统交互 |
| 业务层（Business） | Service 类 | 封装业务规则（BR） |
| 持久层（Data/Persistence） | Repository 类 | 数据存取 |

**分层依赖规则：** 上层可依赖下层，反之不可；同层可相互调用。

### 7. 序列图（Sequence Diagram）

**核心要素：**

| 元素 | 画法 | 说明 |
|:---|:---|:---|
| 生命线 | 对象名:类名 + 向下虚线 | 每个参与者一条 |
| 激活条 | 细长矩形 | 对象正在执行 |
| 同步消息 | ──────→（实线实心箭头） | 调用并等待返回 |
| 异步消息 | ──────→（实线开放箭头） | 调用不等待 |
| 返回消息 | ── ──→（虚线箭头） | 返回值 |
| 自调用 | 从自身发出回到自身 | 内部方法调用 |

**组合片段速查：**

| 片段 | 含义 |
|:---|:---|
| **alt** | 条件分支（互斥），虚线分割 |
| **opt** | 可选执行 |
| **loop** | 循环 |
| **par** | 并行执行 |

**三层对应关系：** 表示层（边界类）→ 业务层（控制类）→ 持久层（实体类）

![序列图示例](blob:https://blogmanager.eightyfor.top/91a0353f-ad11-450c-a459-036c25c648c0)

### 8. 状态图（Statechart Diagram）

**何时创建状态图：**
- 类有复杂的生命周期
- 实例在整个生命周期中可能以多种方式更新属性
- 两个类相互依赖
- 对象的当前行为取决于之前发生的事情

**核心要素：**

| 元素 | 图形 | 说明 |
|:---|:---|:---|
| 初始状态 | ● 实心圆 | 起点 |
| 终止状态 | ◉ 实心圆套空心圆 | 终点 |
| 状态 | 圆角矩形 | 状态名称 |
| 转换 | 箭头 | `事件[守卫条件]/动作` |

**转换标注格式：** `事件名(参数) [守卫条件] / 动作`

**事件类型：** 信号/异步消息、同步调用、时间事件

![状态图示例](blob:https://blogmanager.eightyfor.top/669ec678-e9bf-4a90-9a6a-cf5671f4debd)

### 9. 通信图（Communication Diagram）

- 与序列图表达相同信息，但强调**对象组织**而非时间顺序
- 对象用矩形，关系用连线，消息编号表示顺序
- 适合展示整体对象协作结构

## 三、结构化决策工具

### 10. 决策表（Decision Table）

**四象限结构：**

| 区域 | 内容 |
|:---|:---|
| 左上：条件桩（Condition Stubs） | 所有相关条件 |
| 右上：条件备选（Condition Alternatives） | 每个条件的所有可能取值（Y/N） |
| 左下：动作桩（Action Stubs） | 所有可能的动作 |
| 右下：规则（Rules） | 每条规则对应的动作（X/✓） |

**四大检查：**

| 问题 | 含义 | 处理 |
|:---|:---|:---|
| 不完备（Incompleteness） | 有 N 个条件需 2^N 条规则，遗漏了组合 | 补全或标记"不可能" |
| 不可能情况（Impossible） | 现实中不会出现的条件组合 | 标记"Impossible" |
| 矛盾（Contradiction） | 两条规则条件相同但动作不同 | 修正合并 |
| 冗余（Redundancy） | 两条规则条件相同且动作相同 | 合并为一条，无关条件用"—" |

**简化规则：** 合并条件值不影响结果的行，无关条件用"—"（不关心/Don't Care）表示。

**示例——运费计算规则：**

> 条件：VIP 会员？(Y/N)、订单金额 ≥ ¥500？(Y/N)、加急配送？(Y/N)
> 动作：包邮、¥5 运费、¥10 运费

| 条件桩 | | R1 | R2 | R3 | R4 | R5 | R6 | R7 | R8 |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| 条件 | VIP？ | Y | Y | Y | Y | N | N | N | N |
| 条件 | ≥ ¥500？ | Y | Y | N | N | Y | Y | N | N |
| 条件 | 加急？ | Y | N | Y | N | Y | N | Y | N |
| **动作桩** | | | | | | | | | |
| 动作 | 包邮 | ✓ | ✓ | | | | | | |
| 动作 | ¥5 运费 | | | ✓ | ✓ | | ✓ | | |
| 动作 | ¥10 运费 | | | | | ✓ | | ✓ | ✓ |

**简化后的决策表（合并无关条件）：**

| 条件桩 | | R1 | R2 | R3 | R4 |
|:---|:---|:---|:---|:---|:---|
| 条件 | VIP？ | Y | Y | N | N |
| 条件 | ≥ ¥500？ | Y | N | — | — |
| 条件 | 加急？ | — | — | Y | N |
| **动作桩** | | | | | |
| 动作 | 包邮 | ✓ | | | |
| 动作 | ¥5 运费 | | ✓ | ✓ | |
| 动作 | ¥10 运费 | | | | ✓ |

> 简化说明：R1+R2 合并了 VIP 的前 4 条规则（加急条件无关），R3 合并了非 VIP 订单 ≥500 且加急，R4 合并了非 VIP 且不加急（金额无关）。

**参考：** §09 p18–p23

### 11. 决策树（Decision Tree）

**画法：** 从左到右绘制，根在左、叶在右，每个分支节点标注条件，每个叶子标注结果/动作。

**示例——同一运费规则的决策树（推荐顺序：先判断 VIP→金额→加急）：**

```
                           ┌── 加急？── Y ── [包邮]
                           │
                    ┌─ Y ──┤
                    │      └── 加急？── N ── [包邮]
                    │
           ┌─ Y ───┤
           │       │
           │       │      ┌── 加急？── Y ── [¥5]
           │       │      │
           │       └─ N ──┤
           │              └── 加急？── N ── [¥5]
           │
    VIP？──┤
           │
           │              ┌── 加急？── Y ── [¥10]
           │              │
           │       ┌─ Y ──┤
           │       │      └── 加急？── N ── [¥5]
           │       │
           └─ N ──┤
                   │
                   │      ┌── 加急？── Y ── [¥10]
                   │      │
                   └─ N ──┤
                          └── 加急？── N ── [¥10]
```

**简化版（剪除无关分支，直接标结果）：**

```
                           ┌─ Y ── [包邮]
                    ┌─ Y ──┤
                    │      └─ N ── [包邮]
           ┌─ Y ───┤
           │       │      ┌─ Y ── [¥5]
           │       └─ N ──┤
           │              └─ N ── [¥5]
    VIP？──┤
           │       ┌─ Y ── [¥10]
           │       │
           └─ N ──┤
                   └─ N ── [¥10]
```

> 关键观察：VIP=Y 时加急无影响（都是包邮或 ¥5）；金额=N 时 VIP 无影响（¥5 或 ¥10 取决于加急）。决策树让条件优先级一目了然——不适合用决策表时（条件不完全正交、顺序敏感），用决策树。

**适用场景：**
- 复杂分支且决策顺序至关重要
- 并非每个条件都与每个动作相关
- 需要一目了然地看清条件检查顺序

**vs 决策表的选择：**
- 结构化自然语言：多个重复操作、沟通重要
- 决策表：复杂条件组合、需要避免错误
- 决策树：顺序关键、条件不完全正交

**参考：** §09 p18

### 12. 结构化自然语言（Structured English）

**四种基本结构：**

| 结构 | 语法 | 用途 |
|:---|:---|:---|
| 顺序（Sequential） | Action #1; Action #2; Action #3 | 逐步执行 |
| 决策（Decision） | IF ... THEN ... ELSE ... ENDIF | 条件分支 |
| 分支（Case） | CASE ... WHEN ... WHEN ... ENDCASE | 多路选择 |
| 迭代（Iteration） | DO WHILE ... ENDDO | 循环 |

**适用：** 逻辑涉及公式或迭代且决策不复杂时使用；优势是逻辑清晰、与用户沟通有效。

**参考：** §09 p18

## 四、项目管理工具

### 13. 甘特图（Gantt Chart）

**画法：** 横向柱状图，每行一个任务，横轴=时间，柱长=持续时间，按比例绘制。

**要素：**
- 纵轴：WBS 分解出的任务列表
- 横轴：时间线（天/周/月）
- 横柱：任务起止时间
- 可标注依赖关系和里程碑

**优势：** 直观、适合与最终用户沟通

**参考：** §06 p53

---

### 14. PERT 图（Program Evaluation and Review Technique）

**画法：** 节点表示活动（或箭头表示活动），有向边表示前置关系。从开始到结束的最长路径 = 关键路径。

**核心概念：**

| 概念 | 含义 |
|:---|:---|
| 关键路径（Critical Path） | 决定项目最短工期的路径，无松弛 |
| 松弛时间（Slack Time） | 活动可延迟而不影响总工期的时间 |
| 虚活动（Dummy Activity） | 仅表示逻辑依赖、不消耗时间的伪活动 |

**PERT 表格格式：** 活动名 | 前置活动 | 持续时间

**优势：** 轻松识别优先级顺序、关键路径和关键活动、松弛时间。

**参考：** §06 p53

---

### 15. WBS（工作分解结构，Work Breakdown Structure）

**画法：** 树状/层级图，从顶层项目目标逐级分解到可管理的工作包。

**两种导向：**
- 面向产品（Product-oriented）：按交付物分解
- 面向过程（Process-oriented）：按工作流程分解

**原则：** 每个叶子节点应足够小，可分配给单个人估算和执行。

**参考：** §06 p53

---

## 五、矩阵与表格工具

### 16. CRUD 矩阵

**结构：** 行 = 活动/过程，列 = 数据实体，单元格 = C/R/U/D

| 操作 | 含义 |
|:---|:---|
| **C**reate | 创建新记录 |
| **R**ead | 读取/查询 |
| **U**pdate | 更新现有记录 |
| **D**elete | 删除记录 |

**检查：** 确保每个主文件（Master File）在系统某处有全部四项操作。

**参考：** §08 p25

---

### 17. 事件响应表（Event Response Table）

**结构：** 事件 | 来源 | 触发器 | 活动 | 响应 | 目的地

**核心概念：** 外部实体的输入流 = 触发器，触发 → 活动 → 响应 → 目的地。表格的每一行对应一个 DFD 片段，用于创建单个 Process。

**事件类型：**
- 外部事件（External）：来自外部实体
- 时间事件（Temporal）：基于时间触发

**参考：** §08 p25

---

### 18. 追溯矩阵（Traceability Matrix）

**结构——类→需求矩阵：**

| 类名 | UC01 | UC02 | NF01 | BR01 | 说明 |
|:---|:---|:---|:---|:---|:---|
| Vehicle | ✓ | ✓ | | | 状态变更贯穿全流程 |
| RepairRequest | | ✓ | | ✓ | 核心业务对象 |

**结构——方法一致性矩阵：**

| 方法 | 所在类 | 调用方 | 类图定义 | 序列图使用 |
|:---|:---|:---|:---|:---|

### 19. 数据立方体（Data Cube）— OLAP 操作

```
      产品维
       ↑
       │  ┌──────────────┐
       │ ╱              ╱│
       │╱              ╱ │
       └──────────────┐  │
       │              │  │
       │              │  │
       │              │ ╱
       │              │╱
       └──────────────┘────→ 地区维
                    ╱

                时间维
```

**五大操作：**

| 操作 | 英文 | 说明 |
|:---|:---|:---|
| 切片 | Slice | 固定一个维度的值，提取一个切面 |
| 切块 | Dice | 限定多个维度的范围，提取子立方体 |
| 旋转 | Pivot | 旋转坐标轴，切换视角 |
| 下钻 | Drill Down | 从汇总到明细（年→季→月） |
| 上卷 | Roll Up | 从明细到汇总（城市→省份→国家） |

### 20. 规范化（Normalization）— 三范式

| 范式 | 规则 | 通俗解释 |
|:---|:---|:---|
| 1NF | 原子值，无重复组 | 一个单元格，一个事实 |
| 2NF | 1NF + 无对复合主键的部分依赖 | 不要把两件事混在一张表 |
| 3NF | 2NF + 无传递依赖 | 非主键属性只依赖于主键 |

**三大异常（不规范化的后果）：** 更新异常 / 插入异常 / 删除异常

**原则：** 默认为正确性规范化（3NF）；为性能反规范化；两者都做记录。

**参考：** §12 p6

### 高频手绘检查清单：

- **DFD**：四个符号正确？每个 Process 有输入输出？Data Store 连到 Process？
- **ERD**：实体/属性/主键标注？多重性齐全？M:N 用关联实体破解？
- **用例图**：所有参与者都有关联？include/extend 使用正确？系统边界明确？
- **活动图**：每个决策出口有路径？有开始和结束？泳道分工清晰？
- **类图**：多重性标注齐全？关系类型正确（关联/聚合/组合/泛化）？
- **序列图**：消息接收方有对应方法？alt 分支覆盖所有条件？
- **状态图**：每个状态都有入口和出口？守卫条件明确？
- **决策表**：N 个条件覆盖 2^N 条规则？检查了不完备/矛盾/冗余？
- **甘特图**：按比例绘制？任务起止时间明确？
- **PERT图**：关键路径标注？松弛时间计算？
- **追溯矩阵**：每个需求至少被一个类覆盖？序列图方法在类图中存在？
