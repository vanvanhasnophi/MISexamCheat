# Agent Prompts — 知识点内容与同义词词典生成

> 本文档包含面向外部 AI Agent 的通用 prompt，不绑定具体文件名或路径。
> 每个 prompt 含**有条件输出**指令：Agent 需先判断自己能否写文件，再选择对应的输出方式。

---

## Prompt 1: 生成双语知识点内容

### 背景

用户正在构建一个知识检索 Web App，该 App 按章节和知识点块组织内容，支持中英文切换和同义词搜索。你需要根据用户提供的**源材料附件**，生成两份 Markdown 格式的知识点内容。

### 源材料

用户会在对话中附带一份源材料（如课程 PPT 文本提取、教材笔记等）。请先完整阅读附件内容，理解其结构和主题。

### 输出

生成 **两份内容文件**，一份中文、一份英文，**详细程度完全相同**，仅语言不同：

| 语言 | 风格 |
|:---|:---|
| 中文版 | 详细风格，保留概念细节 + 方法论描述 + 案例梗概 + 完整表格 |
| 英文版 | 同上，仅语言为英文；关键术语标注中文翻译 |

**两份文件的章节结构、知识点块划分、表格、案例必须一一对应**。

### 格式规范

使用以下 Markdown 结构（两份文件遵守同一规范）：

```
# 文档一级标题（可选）

## 01. 章节名称

### 知识点块标题

块正文。使用标准 Markdown：
- 无序列表项
- **粗体** 强调关键术语
- `代码` 包裹技术缩写

表格格式：
| 列1 | 列2 |
|:---|:---|
| 值1 | 值2 |

### 下一个知识点块标题

...

---
## 02. 下一章
...
```

**硬性约束：**

| 规则 | 说明 |
|:---|:---|
| 章节标题仅用 `## NN. ...` | `NN` 为两位数章节编号 |
| 块标题仅用 `### ...` | 不得使用 `####` 或更深层级 |
| 表格必须有 `|:---|` 分隔行 | 至少三个连字符 |
| 正文内不要出现 `##` 或 `###` | 会被解析器误判为边界 |
| 文件编码 UTF-8 无 BOM | 确保多语言正常 |
| 章节间用 `---` 分隔 | 可省略 |

### 内容要求（两份文件一致）

- 保留概念定义和方法论细节，不做过度的精简
- 案例简化为梗概（3~5 句：背景 → 关键问题 → 结果 → 教训）
- 保留图片引用（如 `Image source: example.com`、`FIGURE X.X`）
- 保留完整表格
- 关键术语首次出现时标注双语：中文版标注英文原文，英文版标注中文翻译
- **使用规范化的术语表述**——同一概念在两个版本中使用相同的核心词（如中文版用"业务流程重组"，英文版用"Business Process Reengineering"），确保后续同义词词典能精确匹配

### 输出条件判断

**请先确认你是否具备文件写入能力：**

**→ 如果你能写文件（有 Write / Edit 类工具）：**
1. 先确认用户指定的输出目录和文件名
2. 将中文版写入用户指定的路径（如 `zh.md`）
3. 将英文版写入用户指定的路径（如 `en.md`）
4. 告知用户写入完成

**→ 如果你不能写文件（仅能输出文本）：**
请用以下格式输出，方便用户手动保存：

````
=== FILE: [用户指定的中文文件名] ===
（中文版完整 Markdown）
=== END FILE ===

=== FILE: [用户指定的英文文件名] ===
（英文版完整 Markdown）
=== END FILE ===
````

## Prompt 2: 生成同义词词典

### 背景

同一个 Web App 使用外部同义词词典提供中英文搜索扩展。用户输入任一词时，词典将其扩展为所有等同表述，同时用于搜索匹配和下拉建议。

### 数据来源

从 Prompt 1 生成的**中文版和英文版内容**中提取关键术语。提取时应确保词典中的每个条目都对应内容中实际出现的术语表述。

### 输出格式

严格的 JSON：

```json
{
  "缩写或术语": ["同义词1", "同义词2", "..."],
  "中文全称": ["英文全称", "缩写", "..."]
}
```

**规则：**

| 规则 | 示例 |
|:---|:---|
| 双向显式定义 | `"erp": ["企业资源计划"]` 和 `"企业资源计划": ["erp"]` 各自一条 |
| Key 为小写英文或中文原文 | `"scm"` / `"供应链管理"` |
| Value 列出所有等同表述 | 至少包含中英文各一个 |
| 与内容中使用的术语一致 | 内容写"企业资源计划"，词典就不能写"企业资源规划" |
| JSON 合法可解析 | 无尾逗号、双引号正确 |
| 建议条目数 | 50~120 条 |

### 术语提取范围

从内容中覆盖以下类别：
- 领域缩写（ERP, SCM, CRM, DSS, TPS, ESS, OLAP, OLTP, BPR, TQM...）
- 核心概念（Porter's Five Forces, Value Chain, Bullwhip Effect, Normalization, Digital Firm...）
- 方法论和框架（SDLC, Agile, Scrum, DevOps, UML, DFD, ERD, JAD, STROBE...）
- 技术术语（DBMS, Data Warehouse, Data Mining, OLAP, OLTP, NoSQL, Vector Database, RAG...）
- 管理和组织术语（Best Practices, Change Agent, Disintermediation, Long Tail, Two-sided Market...）

### 输出条件判断

**→ 如果你能写文件：**
写入用户指定的路径（如 `synonyms.json`）

**→ 如果你不能写文件：**
````
=== FILE: [用户指定的词典文件名] ===
（完整 JSON）
=== END FILE ===
````

## Prompt 3: 一键组合生成

如果用户已提供源材料附件，可使用以下组合指令一次完成全部生成：

```
请根据附件中的源材料，依次完成：

1. 按以下 Markdown 格式规范，生成两份详细程度相同但语言不同的知识点内容：
   - 章节以 "## NN. 章节名" 标识
   - 知识点块以 "### 块标题" 标识
   - 表格必须有 |:---| 分隔行
   - 两份文件一一对应：相同章节、相同块划分、相同表格

2. 从生成的内容中提取关键术语，生成 JSON 格式的中英文同义词词典：
   - 双向显式定义（EN→CN 和 CN→EN 各一条）
   - 术语表述与内容中使用的完全一致
   - 50~120 条

3. 判断你是否能写文件：
   - 能写 → 直接写入用户指定的路径
   - 不能写 → 用 === FILE: xxx === ... === END FILE === 格式输出
```

## 附录：格式快速参考

### Markdown 输入规范

```
## 01. Chapter Title

### Block Title

Content with **bold**, *italic*, `code`.

| Header | Header |
|:---|:---|
| Cell | Cell |

### Another Block

...

---
## 02. Next Chapter
...
```

### JSON 词典规范

```json
{
  "it": ["信息技术", "information technology"],
  "信息技术": ["information technology", "it"],
  "erp": ["企业资源计划", "enterprise resource planning"],
  "企业资源计划": ["enterprise resource planning", "erp"],
  "scm": ["供应链管理", "supply chain management"],
  "供应链管理": ["supply chain management", "scm"]
}
```

每对术语需要两条记录（A→B 和 B→A），确保搜索时无论用户输入哪种语言都能命中。
