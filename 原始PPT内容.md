**What is an Information System?**

Information system (IS)
- A set of interrelated components
- Collect, process, store, and distribute information
- Support decision making, coordination, and control
- Help managers and workers analyze problems, visualize complex subjects, and create new products

Example: checkout system in a supermarket
- A set of hardware, software, and people
- Deal with information about products, customers, sales, and payments
- Support decisions about procurement, inventory, and pricing
- Help analyze problem and consider new products

**Data and Information**

- Data are streams of raw facts
- Information is data shaped into meaningful form
- Alternative definition: information is the meaning conveyed in data

**Functions of an Information System**

An information system contains information about an organization and its surrounding environment.

Three basic activities—input, processing, and output—produce the information organizations need.

Feedback is output returned to appropriate people or activities in the organization to evaluate and refine the input.

Environmental actors, such as customers, suppliers, competitors, stockholders, and regulatory agencies, interact with the organization and its information systems.

**What is an Organization?**

- Technical definition
- Formal social structure that processes resources from environment to produce outputs
- A formal legal entity with internal rules and procedures, as well as a social structure
- Behavioral definition
- A collection of rights, privileges, obligations, and responsibilities that is delicately balanced over a period of time through conflict and conflict resolution

**Technical (Microeconomic) Definition of the Organization**

- Inputs: capital and labor (the primary production factors provided by the environment)
- Production process: transform inputs into products and services (outputs to the environment).
- Outputs: The products and services are consumed by the environment, which supplies additional capital and labor as inputs in the feedback loop.

- When technology changes are introduced into the company,
- a technical view of organizations focuses on how inputs are combined to create outputs
- Therefore, from the technical view, building information systems is to
- re-shape the production process
- However, this is incomplete.

**The Behavioral View of Organizations**

The behavioral view of organizations emphasizes group relationships, values, and structures.

- People in an organization develop customary ways of working
- They gain attachments to existing relationships
- They make arrangements about how work will be done
- Most of these arrangements and feelings are not discussed in any formal rulebook.

The behavioral view of organizations emphasizes group relationships, values, and structures.

Building new information systems involves much more than a technical re-arrangement of machines or workers

Information systems may change the organizational balance of rights, privileges, obligations, responsibilities, and feelings

Changing these elements can take a long time, be very disruptive, and requires more resources to support training and learning.

**Transaction Processing Systems (TPS)**

- Serve operational managers and staff
- Perform and record daily routine transactions necessary to conduct business
- Examples: sales order entry, payroll, shipping
- Allow managers to monitor status of operations and relations with external environment
- Serve predefined, structured goals and decision making

Payroll data on master file

**Management Information Systems (MIS)**

- Serve middle management
- Provide reports on firm's current performance, based on data from TPS
- Provide answers to routine questions with predefined procedure for answering them
- Typically have little analytic capability

**Decision Support Systems (DSS)**

- Serve middle management
- Support nonroutine decision making
- Example: What is the impact on production schedule if December sales doubled?
- May use external information as well TPS/MIS data

**Executive Support Systems (ESS)**

- Support senior management
- Address nonroutine decisions
- Requiring judgment, evaluation, and insight
- Incorporate data about external events (e.g., new tax laws or competitors) as well as summarized information from internal MIS and DSS
- Example: Digital dashboard with real-time view of firm's financial performance

**Database Management Systems (DBMS)**

Database
- a collection of data organized to serve many applications efficiently by centralizing the data and controlling redundant data.

Database management system (DBMS)
- software that permits an organization to centralize data, manage them efficiently, and provide access to the stored data by application programs.

Represent data (files) as two-dimensional tables
Each table contains data on entity and attributes
Table: grid of columns and rows
Rows (tuples): Records for different entities
Fields (columns): Represents attribute for entity
Key field: Field used to uniquely identify each record
Primary key: Field in table used for key fields
Foreign key: Primary key used in second table as look-up field to identify records from original table

**Emerging Database Forms**

Non-relational databases: NoSQL
- More flexible data model
- Data sets stored across distributed machines
- Easier to scale
- Handle large volumes of unstructured and structured data

Vector databases (向量数据库)
- Stores unstructured data (text, images, audio) as high-dimensional numerical arrays (embeddings) generated by neural networks.
- Executes semantic similarity search to find data with similar meaning, rather than relying on exact keyword or relational matching.
- Functions as the critical "long-term memory" infrastructure for Generative AI, fundamentally powering Retrieval-Augmented Generation (RAG) architectures.

Databases in the cloud
- Examples: Aliyun RDS, Amazon AWS Database, Microsoft SQL Azure Database
- Benefits: better respond to real-time demand, keep costs low

**Case: Lidl's €500 Million SAP Disaster (2018)**

- Context
- German discount supermarket chain Lidl attempted to replace its legacy inventory-management system with a standard SAP S/4 HANA ERP.
- The project began in 2011, with a planned rollout across 10,000 stores in 29 countries.
- Outcome
- After seven years and €500 million spent, Lidl abandoned the SAP rollout in 2018 and reverted to its legacy system.
- The project is widely cited as a textbook example of "paving the cow path" – attempting to automate broken processes rather than redesigning them.

**How Poor Analysis & Design Caused Failure**

| Phase | Critical Flaw | Consequence |
| :--- | :--- | :--- |
| Analysis | Inadequate requirements understanding: Lidl's business logic calculated inventory at purchase prices; SAP's standard logic uses retail prices. Analysts did not identify this fundamental mismatch during requirements gathering. | The system could not produce accurate financial reports or stock valuations. |
| Design | Refusal to re-engineer business processes: Instead of adapting Lidl's processes to SAP's global best practices (Business Process Re-engineering), the team demanded massive customizations to the core SAP code. | Customizations made the system slow, unstable, and impossible to upgrade. |

**Case: Moutai's (贵州茅台) "Data Middle Platform" (数据中台) Collapse (2020)**

- Context
- Kweichow Moutai (贵州茅台) hired an Alibaba-ecosystem vendor to build a "Data Middle Platform" (数据中台)
- intended to unify data across its distribution network and bypass traditional distributors.
- Outcome
- The project became a notorious industry disaster, with reports of Moutai executives shouting at the vendor to "get out."
- The platform was effectively abandoned, and Moutai returned to its legacy manual-and-Excel-based processes.

**How Poor Analysis & Design Caused Failure**

| Phase | Critical Flaw | Consequence |
| :--- | :--- | :--- |
| Analysis | Ignored organizational culture: Analysts failed to assess the clash between the agile, transparent “internet-style” platform and Moutai’s rigid, hierarchical SOE (State-Owned Enterprise) management style. | The platform was rejected by internal stakeholders who saw it as a threat to established power structures. |
| Design | Over-centralization without fallback: The design assumed full data integration across all business units, with no modular roll-out or contingency for partial adoption. | When key distributors refused to provide data, the entire platform became useless. |
| Stakeholder involvement | Excluded line-of-business managers: Requirements were gathered only from top executives; middle-level managers who actually run daily operations were not consulted. | The system did not support actual workflow needs, leading to workarounds and data fragmentation. |

**Case: MissFresh (每日优鲜) Supply-Chain Algorithm Bankruptcy (2022)**

Context
- MissFresh pioneered the "front-warehouse" (前置仓) model for fresh-grocery delivery,
- relying on sophisticated SCM predictive algorithms
- to push inventory to micro-warehouses in residential neighborhoods for 30-minute delivery.

Outcome
- MissFresh collapsed, leaving thousands of employees unpaid
- illustrating that a "technically perfect" supply-chain system is worthless if its operating costs exceed the product's gross margin.

**How Poor Analysis & Design Caused Failure**

| Phase | Critical Flaw | Consequence |
| :--- | :--- | :--- |
| Analysis | Underestimated demand volatility: Analysts assumed fresh-food demand could be forecast with the same accuracy as packaged goods, ignoring extreme spoilage risks and seasonal spikes. | Algorithms consistently over-stocked perishables, leading to massive waste (spoilage rates >30%). |
| Design | No margin-aware cost modeling: The system design optimized for delivery speed but did not incorporate the true cost of ultra-fast fulfillment (cold-chain logistics, last-mile labor, IT overhead). | Fulfillment costs were 3–6× higher than traditional retail, destroying the profit model. |
| Feasibility study | Technical feasibility prioritized over economic feasibility: The project team proved the algorithm could work technically, but never validated whether the business could afford the required IT and logistics infrastructure. | The company burned through ¥10 billion and filed for bankruptcy in July 2022. |

**Basic Rules**

- The data flow diagram must have at least one process
- Must not be any freestanding objects
- A process must have both an input and output data flow
- A data store must be connected to at least one process
- External entities should not be connected to one another

**Developing Data Flow Diagrams Using a Top-Down Approach**

- Make a list of business activities and use it to determine various
- External entities
- Data flows
- Processes
- Data stores
- Creating the Context Diagram
- Drawing Diagram 0 (The Next Level)
- Creating Child Diagrams (More Detailed Levels)
- Checking Diagrams for Errors
- Develop physical data flow diagram from logical data flow diagram
- Partition the physical data flow diagram

**Creating the Context Diagram**

- The highest level in a data flow diagram
- Contains only one process, representing the entire system
- The process is given the number 0
- All external entities, as well as major data flows are shown

**Diagram 0**

- The explosion of the context diagram
- May include up to nine processes
- Each process is numbered
- Major data stores and all external entities are included

**Data Flow Diagram Levels**

- Data flow diagrams are built in layers
- The top level is the context level
- Each process may explode to a lower level
- The lower level diagram number is the same as the parent process number
- Processes that do not create a child diagram are called primitive

**Creating Child Diagrams**

- Each process on diagram 0 may be exploded to create a child diagram
- A child diagram cannot produce output or receive input that the parent process does not also produce or receive
- The child diagram is given the same number as the process in the parent diagram
- Process 3 would explode to Diagram 3
- Entities are usually not shown on the child diagrams below Diagram 0
- If the parent process has data flow connecting to a data store, the child diagram may include the data store as well
- When a process is not exploded, it is called a primitive process

**From Logical to Physical**

- Derive the logical data flow diagram for the current system by examining the physical data flow diagram and isolating unique business activities.
- Create the logical data flow diagram for the new system by adding the input, output, and processes required in the new system to the logical data flow diagram for the current system.
- Derive the physical data flow diagram by examining processes on the new logical diagram. Determine where the user interfaces should exist, the nature of the processes, and necessary data stores.

- Validation processes for ensuring accurate data input
- Sequencing processes to rearrange the order of records
- Processes to produce every unique system output
- Intermediate data stores
- Actual file names used to store data
- Controls to signify completion of tasks or error conditions

**Logic Descript (3): Decision Trees (决策树)**

- Decision trees are used when complex branching occurs in a structured decision process
- Trees are also useful when it is essential to keep a string of decisions in a particular sequence
- Drawing decision trees
- Identify all conditions and actions and their order and timing (if they are critical)
- Begin building the tree from left to right, making sure you list all possible alternatives before moving to the right

**Decision Tree Advantages**

- The order of checking conditions and executing actions is immediately noticeable
- Conditions and actions of decision trees are found on some branches but not on others
- Compared to decision tables, decision trees are more readily understood by others in the organization

**Selecting a Structured Decision Analysis Technique**

- Use structured English when there are many repetitious actions or when communication to end users is important
- Use decision tables when a complex combination of conditions, actions, and rules are found or you require a method that effectively avoids impossible situations, redundancies, and contradictions
- Use decision trees when the sequence of conditions and actions is critical or when not every condition is relevant to every action (the branches are different)

**Normal Forms**

| Form | Rule | Plain Explanation |
| :--- | :--- | :--- |
| 1NF | Atomic values, no repeating groups | One cell, one fact |
| 2NF | 1NF + no partial dependency on composite key | Don't mix two things in one table |
| 3NF | 2NF + no transitive dependency | Non-key attributes depend on the key only |

- Practice:
- Aim for 3NF by default
- Denormalize INTENTIONALLY when query performance demands it
- Normalize for correctness
- Denormalize for performance
- Document both

**Vector Databases (向量数据库)**

- Core insight: Store MEANING as COORDINATES, not text.
- Imagine a 2-D "meaning map":
"Apple iPhone" (90, 20) | almost identical!
"Huawei phone" (88, 22) |
"Apple juice" (15, 80) - far away
- "Apple iPhone" and "Apple juice" share a WORD but live FAR APART.
- "Apple iPhone" and "Huawei phone" share NO WORDS but live RIGHT NEXT to each other.
- A neural network (the "embedding model") learns these coordinates by reading the entire Internet.
- Traditional DB: find books by ISBN NUMBER
- Vector DB: find books by THEIR POSITION ON A MAP

**Embeddings: From Meaning to Math**

- With vectors, "similar meaning" is a measurable quantity.
Real systems typically use 1536 dimensions.

"农村在线辅导" → [0.21, -0.83, 0.45, ..., 0.19] (1536 numbers)
"偏远地区教育" → [0.19, -0.81, 0.48, ..., 0.22] ← cosine similarity = 0.92
"披萨外卖" → [-0.55, 0.71, -0.32, ..., 0.81] ← cosine similarity = 0.05

Embedding model = a neural net that maps text / image / audio into these coordinates, trained so that SIMILAR MEANING → CLOSE DISTANCE.
- OpenAI text-embedding-3-small → 1536 dims
- BGE, M3E, Sentence-BERT → Chinese-friendly OSS alternatives
- CLIP (text+image), Whisper (audio) → same idea, different inputs

Tools: Pinecone (cloud), Weaviate / Milvus (OSS), ChromaDB (local), pgvector (in Postgres)

**Agentic Interface**

The user expresses a GOAL
The system PLANS and EXECUTES multiple steps, asking the user only at critical checkpoints.

Examples in 2026:
- Cursor / Claude Code / OpenCode: "fix this bug end-to-end"
- Manus / OpenClaw / Hermes: "book my Shanghai trip"

It is NOT just "natural language":
- Natural Language Query: system returns ONE answer; you decide what to do next
- Agentic: system runs a multi-step PLAN; you supervise, not operate

What it requires that previous paradigms did not:
- Tools the agent can call (APIs, code execution, browsing, file edits)
- Memory across many turns (so the agent does not forget mid-task)
- Human-in-the-loop checkpoints (approve, interrupt, replay, audit)

For the first time, the user does not OPERATE the system. The user DELEGATES to the system.

**Purposes of Codes**

Meaningful codes (有业务意义的编码)

**Effective Coding**

- Data that are coded require less time to enter
- Coding helps to reduce the number of items entered
- Coding can help in sorting data during the data transformation process
- Coded data can save valuable memory and storage space

**Human Purposes for Coding**

- Keeps track of something
- Classifies information
- Conceals information
- Reveals information
- Requests appropriate action

**Simple Sequence Code**

- A number that is assigned to something if it needs to be numbered
- No relation to the data itself

| Order # | Product | Customer |
| :--- | :--- | :--- |

**Block Sequence Code**

- An extension of the sequence code
- Data are grouped according to common characteristics

**Alphabetic Derivation Code**

- A commonly used approach in identifying an account number

**Significant-Digit Subset Code**

- Used to help describe a product by virtue of its membership in many subgroups
- Possible to locate items that belong to a certain group or class
- Inquiries may be performed on a portion of the code

**Other Codes**

- Classification Code (分类码)
- Cipher Code (密码)
- Mnemonic Code (助记码)
- Function Code (功能码)

**General Guidelines for Coding**

- Be concise
- Keep the codes stable
- Ensure that codes are unique
- Allow codes to be sortable
- Avoid confusing codes
- Keep the codes uniform
- Allow for modification of codes
- Make codes meaningful

**Coding Example**

- sku: significant-digit subset
- customer_id: simple sequence
- order_id: block sequence

**Why Meaningful for SKU**

Because products are often operated on by classification.
For example
[category]-[model]-[color]-[specification] LM-X3-W-STD

"All speakers get 20% off for Double 11."
UPDATE price SET discount = 0.8 WHERE sku LIKE 'LM%';

Without the code:
UPDATE price SET discount = 0.8 WHERE sku IN (SELECT sku FROM product_category_map WHERE category = 'Audio');

**Significant-Digit Subset Code: the Trade-off**

e.g., [category]-[model]-[color]-[specification] LM-X3-W-STD

| Gain | Cost |
| :--- | :--- |
| Human-readable at a glance | Segment meanings can go stale — if "LM" is renamed to "AU", every SKU must change |
| Sortable by any segment | Space limits per segment — only 26²=676 two-letter combinations |
| Bulk operations in one line of SQL | Violates "Keep Codes Stable" — the richer the semantics, the harder to modify |

**Why Simple Sequence for customer_id**

Because a person's identifier should carry no story.
e.g., 00000847
Display as C-00000847 when needed. Store as a bare integer. No segments. No semantics. No story. Just identity.

"A customer requests account deletion."
With Simple Sequence (surrogate key): UPDATE customers SET name = '[DELETED]', phone = NULL, email = NULL WHERE customer_id = 00000847; PII erased. Orders preserved for audit. Key never changes.
With phone number as PK: DELETE FROM customers WHERE phone = '138xxxx5678'; ➔ Cascades through 87 tables - orders, payments, logistics, reviews... Financial records break. Audit trail destroyed. System integrity fails.

**Simple Sequence Code: the Trade-off**

e.g., 00000847
Display as C-00000847 when needed. Store as a bare integer.
No segments. No semantics. No story. Just identity.

| Gain | Cost |
| :--- | :--- |
| Stable forever — nothing to go stale | Not human-readable — you need a lookup to know who 00000847 is |
| Privacy-safe — the key leaks nothing | Exposes scale — competitor can estimate your customer count |
| Deletable personal information without breaking references | Requires a separate display name/number for customer-facing contexts |

**Why Block Sequence for order_id**

- Because orders are events and may be classified
- e.g., [type]-[date]-[sequence]
- ORD-20260328-001847
- "Pull all ordinary orders from May 31, 2026 for the operations review."
SELECT * FROM orders WHERE order_id BETWEEN 'ORD-20260531-000000' AND 'ORD-20260531-999999';

**Block Sequence Code: Trade-off**

e.g., [type]-[date]-[sequence] ORD-20260328-001847

| Gain | Cost |
| :--- | :--- |
| Naturally time-ordered — index stays clean | Within a day, counter is centralized — sharding needs care |
| Queries by date range = PK range scan | Reveals approximate daily volume to anyone who sees two IDs from the same day |
| Operators recognize timestamps at a glance | Slightly longer than pure counters (18 chars vs 8) |
| Trivial table partitioning by month/year | Cross-day operations still need a date index |

**The Coding Decision: Summary**

- sku: significant-digit subset
- customer_id: simple sequence
- order_id: block sequence

**Effective and Efficient Data Capture**

- The Best Data Entry is No Data Entry
- The Core Principle
- Enter what only humans can know; let the computer figure out the rest.
- A base data item should be entered only once at only one point.

| Question | What to Enter | What NOT to Enter |
| :--- | :--- | :--- |
| Is it changeable? | YES → Must capture each time (quantity, date, customer choice) | NO → Look it up from master data |
| Does it differentiate this record? | YES → Capture (the SKU being ordered) | NO → Derive it (the SKU's name, price, weight) |
| Can the computer compute it? | NO → Capture | YES → Generate it (timestamp, ID, total) |

**Example: an Order Form**

Bad Form - 5 human-typed fields
- Customer Name: ← re-typed every order
- Customer Phone: ← already logged in!
- Product Name: ← type "Lumina-X3 Speaker"?
- Unit Price: ← customer doesn't know this
- Total Amount: ← customer calculates?

Good Form - 1 human-typed field
- Customer: [Auto-filled from login]
- Product: [search sku] → system fills name, price, weight...
- Quantity: ← the ONLY field a human enters
- Total: [Auto-calculated]

Emerging data-entry methods
- OCR: auto-fill address;
- Computer vision: shelf recognition;
- LLM: extract data from unstructured emails;
- IoT sensor: auto-report stock level;
- ...

**Types of Input Validation**

- Missing Data
Is a required field left empty? e.g., customer_id must not be NULL
- Field Length
Is the value the expected length? e.g., Zip code = exactly 6 digits
- Class / Composition
Is it the right character type? e.g., Name field: letters only, no digits
- Range / Reasonableness
Is the value within a plausible range? e.g., Order quantity: 1-10,000; month: 1-12
- Reasonableness
Is the value on a blacklist? e.g., Order status must be P/S/D, not 'X'
- Comparison with Stored Data
Does the foreign key exist in master data? e.g., SKU must exist in Products table
- Self-Checking Digits
Does the embedded check digit verify? (Luhn) e.g., Credit card: last digit validates the rest
- Pattern Match (Regex)
Does the value match a format pattern? e.g., Email: x@x.x | Phone: (ddd) ddd-dddd

**Validation: A Pipeline, Not a Checkpoint**

The Validation Pipeline

Two Layers of Defense

Client-Side Validation (Browser / App)
- Instant feedback — user sees errors before submit
- Catches honest mistakes (typos, wrong format)
- Improves UX — no frustrating round-trips
- Can be bypassed (dev tools, API calls)

Server-Side Validation (The Final Defense)
- Never trust client data — re-validate everything
- Enforces business rules (range, stored data, check digits)
- Protects against malicious input (SQL injection, etc.)
- The only validation that truly matters

**Taming Complexity: Modularity & Documentation**

- Divide and conquer — the oldest engineering principle, still the best.

Why Must Complex Systems Be Modular?
- Human Cognition: No one can hold 100,000 lines of code in their head. Modules let each person understand one piece at a time.
- Parallel Work: Teams work on different modules simultaneously — without stepping on each other's toes.
- Change Isolation: Fix or replace one module without touching the rest. Change is local, not systemic.

The Top-Down Approach
- Start from the whole ⇒ break into subsystems ⇒ break into modules ⇒ each module does ONE thing

**Designing for Maintenance**

- Pay particular attention to the critical interfaces
- Minimize the number of modules the user must modify when making changes
- Maintain the hierarchical relationships set up in the top-down phases
